# Задача для Claude Code: UPC-пул Speedy Barcode → движок создания листингов в SSCC

## Контекст (что уже сделано)

Джеки (агент-оркестратор) провёл аудит штрихкодов Владимира, купленных в сервисе **Speedy Barcodes**, и подготовил пул свободных UPC для создания новых листингов на Amazon/Walmart через командный центр **SSCC** (`salutemsolutions.info`).

### Что именно было проверено (и насколько можно доверять)

1. **Собраны все купленные UPC.** Из писем (Sales@speedybarcodes.com, ящики kuzy.vladimir@ и ancienmadina2@) + папок Google Drive извлечены списки из 14 заказов Speedy Barcodes → **22 810 уникальных купленных UPC** (все с валидной контрольной цифрой UPC-A).

2. **Определены уже занятые.** Весь каталог **Veeqo** (наш товарный хаб, 15 214 товаров) прогнан на предмет присвоенных UPC. Veeqo покрывает подключённые каналы: **5 Amazon-аккаунтов** (Retailer, Salutem, AMZ Commerce, KVV, SIRIUS) + **1 Walmart** (SIRIUS TRADING) + eBay/TikTok. Занятыми оказались **9 571** из купленных.

3. **Свободный пул = 22 810 − 9 571 = 13 239 UPC.** Это и есть содержимое пула в этом каталоге.

### ГРАНИЦЫ ПРОВЕРКИ (важно — не выдавать за 100%)

- Проверка «свободности» сделана **только против нашего Veeqo**. Она НЕ гарантирует, что UPC свободен на маркетплейсе глобально, потому что:
  - **2 из 3 исторических Walmart-аккаунтов НЕ подключены к Veeqo** → листинги оттуда невидимы; их UPC могут быть заняты.
  - Баркоды Speedy — **переиспользуемые GS1-коды**; сервис мог перепродать тот же код другому продавцу, и он уже может висеть на чужом ASIN/листинге.
- **Доказанный пример грязного кода:** текущий пул SSCC на префиксе `742259` — ГРЯЗНЫЙ. Код `742259000034` уже занят на Amazon чужим ASIN **BOH75VN18Z** («800W Battery Charger»). При попытке публикации Amazon вернул ошибку **8541** (`standard_product_id conflicts with ASIN in catalog`). Поэтому **пул 742259 использовать нельзя — заменить на этот, Speedy.**

**Вывод:** 13 239 — это «свободные по нашим данным, но истину скажет сам маркетплейс при публикации». Отсюда — алгоритм ниже с обратной связью.

## Файлы пула (в этом каталоге)

- `speedy_free_pool.csv` — канонический, для импорта в БД. Колонки:
  `upc, state, source_order, check_digit_valid, assigned_sku, marketplace_status, last_updated`
- `speedy_free_pool.jsonl` — то же, по одной JSON-записи на строку (стриминг/импорт). Поля:
  `upc, state, source_order, check_digit_valid, assigned_sku, assigned_asin, marketplace_status, attempts, last_updated`
- `pool_meta.json` — метаданные (кол-во, разбивка по префиксам, словарь состояний).

**Состояния (`state`):**
- `available` — не присвоен, можно брать.
- `assigned` — успешно закреплён за созданным листингом (SKU/ASIN записан).
- `burned` — маркетплейс отверг (занят кем-то другим). НЕ использовать повторно.

## Что нужно сделать (задача Claude Code)

### 1. Завести пул как управляемую таблицу в БД SSCC
- Создать таблицу, напр. `upc_pool_speedy` со схемой из полей выше + уникальный индекс по `upc`.
- Импортировать `speedy_free_pool.csv` (13 239 строк) со `state='available'`.
- **Переключить движок создания листингов SSCC с грязного пула `742259` на этот пул `speedy`.** Старый пул пометить deprecated/не выдавать.

### 2. Реализовать выдачу баркода «под листинг» (atomic)
- Функция `reserve_next_upc(sku)` — атомарно берёт первый `available` (напр. `SELECT ... FOR UPDATE SKIP LOCKED` / транзакция), ставит `state='reserved'`, привязывает `assigned_sku`, инкрементит `attempts`, пишет `last_updated`. Возвращает UPC.
- Порядок выдачи — по возрастанию (стабильный), чтобы не было гонок и дублей.

### 3. Цикл публикации с обратной связью от маркетплейса
Для каждого нового листинга:
1. `upc = reserve_next_upc(sku)`
2. Вставить `upc` в External Product ID листинга, опубликовать на маркетплейс (Amazon/Walmart) через существующий SSCC-флоу.
3. Разобрать ответ:
   - **УСПЕХ** (листинг создан / ASIN присвоен) → `state='assigned'`, записать `assigned_asin`/item_id, `marketplace_status='ok'`. Готово.
   - **ОТКАЗ «код занят»** → пометить `state='burned'`, `marketplace_status=<код ошибки>`, и **сразу взять следующий available и повторить публикацию** (цикл), пока не пройдёт.
   - **Прочая ошибка** (не про UPC — напр. бренд/картинка/габариты) → НЕ жечь баркод; вернуть `state='available'` (release), залогировать причину, эскалировать (это не про баркод).

**Сигнатуры отказа «код занят», которые надо ловить:**
- Amazon: ошибка **8541** («We found more than one ASIN matching…», `standard_product_id conflicts with … ASIN in the catalog»`); также «UPC/EAN already used», «product already exists», GTIN-mismatch.
- Walmart: GTIN/UPC already exists / item already set up / duplicate GTIN.
- **Отличать** «код занят» от «нельзя продавать бренд / gating / невалидные атрибуты» — второе НЕ повод жечь баркод.

### 4. Требования к реализации
- **Идемпотентность и резюмируемость:** процесс можно останавливать/перезапускать без потери и без двойной выдачи (состояние в БД — источник правды).
- **Логирование:** каждая попытка — строка лога (upc, sku, результат, код ошибки, время).
- **Rate limiting / ретраи** под лимиты Amazon SP-API и Walmart API; экспоненциальный backoff на 429/5xx.
- **Метрики/отчёт:** сколько available / assigned / burned; burn-rate (доля занятых) — по нему поймём, насколько пул «чистый» на практике.
- Не хардкодить секреты — брать креды из конфигурации SSCC.

### 5. Ожидаемый результат
Движок SSCC при создании листинга сам берёт следующий чистый UPC из пула Speedy, публикует, и по ответу маркетплейса либо закрепляет код за листингом (`assigned`), либо жжёт (`burned`) и берёт следующий — циклично, автономно, до успеха. Пул из 13 239 самоочищается практикой.
