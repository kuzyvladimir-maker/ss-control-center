# 📚 Reference Catalog (Donor DB) + Engine Parser — архитектура

> Спецификация по итогам разбора с Владимиром (2026-06-19). Это эволюция
> [Product Sourcing Engine](product-sourcing-engine.md): движок Engine Parser
> наполняет **донорскую (справочную) базу**, из которой питаются все модули.
> Связано: `project_reference_catalog`, `reference_buying_retailers` (память).

## 🗄️ ДВЕ базы данных

**1. НАШ КАТАЛОГ** (уже есть) — наши листинги/инвентарь на Amazon/Walmart (что продаём).
`SkuShippingData` + каталоги маркетплейсов. Пример: «томатная паста, 6 единиц в листинге».

**2. ДОНОРСКАЯ / СПРАВОЧНАЯ база** (строим, **product-centric**) — каталог отдельных
реальных единиц товара, собранных парсером с магазинов-доноров. НЕ привязана к нашим SKU —
это «знание о товарах вообще», переиспользуемое всеми модулями. UI-модуль = **Reference Catalog**.

### Модель данных (Prisma → Turso; НЕ markdown/excel)
- **`DonorProduct`** — 1 строка = 1 реальный товар. Поля: brand, product_line, flavor,
  **size + unitMeasure** (oz/count — для сравнения retail vs wholesale), container_type, UPC/GTIN
  (если настоящий), **mainImageUrl** (для таблицы), imageUrls[] (≥5: главное, состав, **nutrition
  label**, инфографика), title, description, bullets[], attributes(JSON), **nutritionFacts**,
  **ingredients**, **pricePerMeasure** (цена за oz — нормализация), createdAt/updatedAt.
- **`DonorOffer`** — цена по каждому магазину для этого товара: retailer, retailerProductId,
  price (за единицу), currency, **via** (`direct` | `instacart`), **zip**, inStock, productUrl,
  sellerName, isFirstParty, fetchedAt. Идемпотентно по (retailer, retailerProductId).
- **`EnrichmentJob`** — очередь «что обогатить»: target (brand | product | sku), status
  (queued/running/done/error), source (cron | manual | bundle-factory), createdAt, result.

**Себестоимость (COGS) = самая дешёвая ЧИСТАЯ DIRECT-цена** среди магазинов, где реально
закупаемся. Instacart-цены — отдельно (см. наценку ниже), не как «дешёвая 1P».

## 📍 ZIP / зона закупки
**Clearwater, Florida 33765** (наш склад). Все цены/наличие тянем под этот zip — храним только
то, что физически можем купить в нашей зоне.

## 🏪 Магазины — приоритет (по обороту/чистоте закупа, Владимир 2026-06-19)
~80% продаж = **сушняк, перепродаём с Walmart и там же закупаем**. Заморозка ~50/50 Walmart и BJ's.
Порядок трат/частоты закупа:
1. **Walmart** — №1, основной (весь dry). → BlueCart ✅ оплачен.
2-3. **Publix** + **BJ's** — паритетно (BJ's особенно заморозка). → нет прямого API (см. ниже).
4. **Target** → Unwrangle.
5. **Sam's Club** → Unwrangle.
6. **Costco** — редко. → Unwrangle.
Редкие/edge: **Amazon.com** (что продаёт сам Amazon), **Whole Foods**, **Winn-Dixie**, **Aldi**,
**Bravo** (Hispanic grocery, FL). Берём, когда нет в основных.

### ⚠️ Проблема Publix/BJ's (tier-1, но трудные)
Самые важные после Walmart, но **прямого API нет**. Варианты: Instacart (агрегатор, с наценкой —
см. ниже) ИЛИ прямой скрейп через Oxylabs. Для ИСТИННОЙ себестоимости приоритет — прямой источник.

### ⚠️ Наценка Instacart
Instacart накручивает ~**15%** поверх цены ритейлера (Target через Instacart ≈ +15% к Target.com
напрямую; коэффициент зависит от магазина). Поэтому: (1) предпочитаем DIRECT-цену; (2) если только
Instacart — храним `via=instacart` + флаг наценки, не считаем её «чистой 1P-себестоимостью».

### ⚠️ Объём единицы (retail vs wholesale)
Restaurant Depot / Gordon Food Service (GFS) продают ТЕ ЖЕ товары, но большими объёмами (томат-паста
retail 8/15 oz vs опт 32/64/128 oz). **Размер/вес единицы обязателен** в каталоге + нормализуем в
**цену за oz** (`pricePerMeasure`), чтобы корректно сравнивать retail и опт.

## 🔒 Контроль качества (жёсткий, 100%)
Только **оригинальный продавец** по каждому источнику (Target-оффер = sold by Target, чтобы реально
заказать онлайн и получить от Target, не от перекупа). `isFirstParty()` обобщить на все ритейлеры.

## 🖥️ Модуль Reference Catalog (страница в Command Center)
- Таблица товаров с **титульной фотографией спереди** (для визуального чекапа: фото = названию,
  товар лицом, не перепутан).
- Фильтры: цена, бренд, размер/граммы, название, магазин, категория.
- **Счётчик уникальных товаров** (всего + по фильтрам).
- **График роста** — сколько новых уникальных товаров парсер добавляет за каждый проход (растущая линия).
- **Поле «вектор» + Run**: написать «обогати по бренду Jimmy Dean / Uncrustables» → задача в `EnrichmentJob`.

## ⚙️ Очередь обогащения — 1 воркер, 3 производителя
Паттерн как у Walmart remediation (Optimizer queue, разгребается Vercel-кроном). Переиспользуем.
- Производители задач: (1) **крон** (авто, ежедневно, несколько проходов в рамках лимитов сервисов);
  (2) **кнопка «вектор»** на странице; (3) **Bundle Factory** (когда нужен товар, которого нет в базе).
- Воркер (крон) разгребает очередь → Engine Parser → наполняет DonorProduct/DonorOffer.

## 🔌 Потребители донорской базы
1. **Улучшение наших листингов** — наш товар → найти в донорской базе → AI переписывает фото/описание/
   буллеты → вернуть в ASIN / Walmart-SKU (рост продаж).
2. **Bundle Factory** — собирать новые наборы/бандлы/киты ИЗ донорской базы (сперва читает её; если
   товара нет → ставит EnrichmentJob → потом собирает). Для Amazon/Walmart/eBay/своего сайта.
3. **Экономика** — реальная себестоимость каждой единицы → репрайсер (margin floor) + чистая прибыль.

## 🪜 Этапы
| # | Этап |
|---|---|
| 0 | Схема: `DonorProduct` + `DonorOffer` + `EnrichmentJob` (+ size/unitMeasure/pricePerMeasure, via, zip) |
| 1 | ⭐ Все платные сервисы: Target/Sam's/Costco (Unwrangle), Publix/BJ's/Aldi/Winn-Dixie/Bravo (Instacart/Oxylabs), zip-aware |
| 2 | Жёсткие 1P-гейты по каждому источнику + Instacart-наценка + dedup |
| 3 | Полный контент-харвест (≥5 фото + nutrition + ingredients + атрибуты) |
| 4 | Наполнение донорской базы + дедуп (1 товар = 1 строка) |
| 5 | Очередь `EnrichmentJob` + воркер-крон + 3 производителя |
| 6 | Модуль Reference Catalog (таблица+фото, фильтры, счётчик, график, поле «вектор») |
| 7 | Потребители: улучшение листингов / Bundle Factory из базы / экономика |

Старт: этап 0 (схема) + этап 1 (платные сервисы) — почти параллельно, это фундамент.
