# 🔎 Walmart Catalog Cache — быстрый поиск SKU для Джеки

## Суть
Локальное зеркало всего каталога товаров Walmart, обновляемое ночным кроном. Существует ради одной проблемы: **у Walmart Marketplace API нет текстового поиска по товарам продавца.**

`GET /v3/items` умеет только две вещи:
1. Вернуть весь список товаров постранично (offset + limit, по 200 на страницу).
2. Вернуть один товар по **точному** SKU (`?sku=...`).

Поиска вида «найди все SKU, в названии которых есть Arnold Potato Buns» — нет. Поэтому раньше `walmart_items_search` (инструмент Джеки) вынужден был **листать весь каталог вживую** (~5000 товаров, ~25 последовательных запросов, **40–60 секунд**) и фильтровать у себя в памяти.

Теперь полный каталог зеркалируется в таблицу `WalmartCatalogItem`, а поиск Джеки читает эту таблицу — **~230 мс вместо 40–60 с** (замерено на проде 2026-05-29).

## ⚠️ Важное уточнение (была путаница)
Старый `walmart_items_search` **не** читал из локальной базы — он каждый раз ходил **напрямую в Walmart API**. Медленно было именно из-за ~25 сетевых запросов в Walmart, а не из-за чтения базы. Починка — это как раз *добавить* локальный кэш, а не «попросить Walmart искать по словам» (Walmart так не умеет).

| Компонент | Файл | Роль |
|-----------|------|------|
| Модель БД | `prisma/schema.prisma` → `WalmartCatalogItem` | sku, itemId, title, lifecycleStatus, publishedStatus, syncedAt. Unique `(storeIndex, sku)` |
| Миграция | `prisma/migrations/20260529010000_walmart_catalog_cache/` | CREATE TABLE + индексы |
| Пагинатор | `src/lib/walmart/items.ts` → `iterateWalmartCatalog()` | листает ВЕСЬ каталог (все publishedStatus) |
| Синк | `src/lib/walmart/catalog-cache.ts` → `syncWalmartCatalog()` | буферит всё в память, затем в одной транзакции `deleteMany` + `createMany` чанками по 500 |
| Поиск | `src/lib/walmart/catalog-cache.ts` → `searchWalmartCatalogCache()` | подстрочный поиск (SKU ИЛИ title) по таблице, по умолчанию только PUBLISHED |
| Ночной крон | `src/app/api/cron/walmart/route.ts` → `syncCatalog` | под-задача в 06:00 UTC, `maxDuration = 300` |
| Инструменты Джеки | `src/lib/jackie-mcp/tools/walmart-inventory.ts` | `walmart_items_search` (читает кэш), `walmart_catalog_refresh` (ручной синк) |

## Как работает синк (надёжность)
1. Листаем весь каталог Walmart → буфер в памяти (~3800 строк с SKU, после дедупа).
2. В **одной транзакции**: удаляем все строки этого аккаунта → вставляем новые чанками по 500 (`INSERT_CHUNK`), таймаут транзакции поднят до 120с (`TX_TIMEOUT_MS`), потому что рантайм-БД — **Turso (remote libsql)**, каждый statement = round-trip по сети.
3. Если пагинация Walmart упала — БД вообще не трогаем. Если упала транзакция — откат, старое зеркало целое. То есть провал синка = «данные слегка устарели», но никогда не «пусто/битое».

## Поведение поиска (`walmart_items_search`)
- Кэш пуст (ни разу не синкался) → fallback на старый медленный живой скан + подсказка запустить крон.
- Кэш есть → читаем из таблицы (быстро). По умолчанию только PUBLISHED; `include_unpublished=true` снимает фильтр.
- 0 совпадений + запрос без пробелов (похож на SKU) → **подстраховка**: один живой запрос `?sku=` в Walmart, чтобы поймать свежесозданный SKU, которого ещё нет в ночном зеркале.

## Что НЕ кэшируется
**Остатки (inventory) пишутся напрямую в Walmart в реальном времени** через `walmart_inventory_update`. Кэшируется только сопоставление «название → SKU», поэтому отставание зеркала на несколько часов безопасно (названия и состав каталога меняются редко).
