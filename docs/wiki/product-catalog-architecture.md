# 🏛️ Архитектура каталога товаров (канон)

> **Зачем этот документ:** единый ответ на «как правильно построить решение»,
> который **прекращает круги** между тремя параллельными чатами (картинки / COGS /
> создание листингов). Сверено с живой схемой `ss-control-center/prisma/schema.prisma`
> (2026-07-05). Все три чата равняются на ЭТОТ документ. Связано:
> [[product-sourcing-engine]], [[reference-catalog-engine]], [[walmart-ideal-listing-spec]],
> [[task-registry]]. Память: `project_reference_catalog`, `project_cogs_catalog_pricing_roadmap`,
> `reference_retail_source_capabilities`.

## 1. Корень кругов — одним предложением

**Три чата сталкиваются, потому что НИ ОДНА таблица не владеет «правдой о товаре»:**
каждый чат завёл свою таблицу товара со своим ключом и своим кодом добычи, поэтому
они не видят результатов друг друга — и каждый заново тянет те же данные, упирается
в ту же стену (бот-блок Walmart, таймаут Unwrangle, UPC-мультипак-ловушка) и падает
одинаково.

Доказательство — в самой схеме: **одни и те же 4 факта** (title, все фото, весь
текст, цена) независимо хранятся и независимо наполняются в **7 таблицах** —
`RetailPrice`, `DonorProduct`, `WalmartCatalogItem`, `AmazonListingSnapshot`,
`GeneratedContent`, `ChannelSKU`, `MasterBundle` — **без общего ключа** (кто по `sku`,
кто по `upc`, кто по `identityKey`, кто по `asin`). Семь копий, семь добытчиков, семь стен.

## 2. Архитектура — одна картинка

```
                         ┌───────────────────────────────┐
   ВНЕШНИЙ МИР           │   КАТАЛОГ (источник правды)    │
 (Walmart, Amazon,       │   ключ = GTIN-14               │
  Target, Unwrangle,     │   Product ──< Sku ──< Listing  │
  SP-API, браузер)       │   3 полки: RAW / OURS / GENER. │
        │                └───────────────────────────────┘
        │                     ▲ ПИШЕТ          ЧИТАЮТ ▲ ▲ ▲
   ┌────┴──────────────────┐  │ (единственный путь)  │ │ │
   │  enrichProduct()      │──┘        ┌─────────────┘ │ └────────┐
   │  ЕДИНСТВЕННЫЙ ДВИЖОК   │       картинки        COGS/    создание
   │  - только он ходит     │      (этот чат)      цены     листингов
   │    во внешний мир      │
   │  - кэширует всё         │   ← ПОТРЕБИТЕЛИ: только ЧИТАЮТ каталог,
   │  - одна очередь, дедуп  │     во внешний мир НЕ ходят
   └────────────────────────┘
```

**Идентичность товара = GTIN-14 (нормализованный штрихкод), а НЕ SKU.** SKU описывает
*пак, который ты продаёшь* («pack of 6»); GTIN — *реальную физическую вещь* (одну булку).
Твои «pack of 6» и «pack of 2» — это два SKU **одного** товара, поэтому фото/текст/цена
хранятся **один раз**, и каждый пак их наследует. **Это и есть твоя проблема
quantity-confusion, решённая в модели данных:** один товар — много размеров пака.

**Как SKU ложатся на товары — трёхуровневый хребет:**
```
Product   (1 строка на реальную вещь, ключ = GTIN-14)   ← идентичность / «булка»
  ▲ много SKU → один Product
Sku       (1 строка на пак, который продаём: "BREAD-6") ← что продаём
  ▲ один SKU → много листингов
Listing   (1 строка на SKU × канал × аккаунт)           ← где живёт (ASIN / walmart itemId)
```
Два типа Product: **BASE** (реальный товар, GTIN производителя) и **ASSEMBLY** (твой
набор/gift-set, внутренний GTIN из `UPCPool`; его данные **собираются из GTIN
компонентов**, не добываются заново).

## 3. Модель данных — 3 слоя на существующих таблицах (не переписываем)

Твоя интуиция (сырьё / наше / сгенерированное) — ровно верна. Каждый слой — свои
таблицы, каждая строка **со штампом источник + время + confidence**. Слои друг друга
не перезаписывают.

**СЛОЙ 1 — RAW («что говорит мир»)** → расширить `DonorProduct` + `DonorOffer`, влить `RetailPrice`.
Единственные таблицы, куда пишет движок; единственное место, куда падают внешние данные.
- `DonorProduct` (ЕСТЬ, расширить) — 1 строка на товар. Уже есть `upc`, `gtin`,
  `identityKey`, `title`, `description`, `bullets`, `attributes`, `imageUrls`,
  `nutritionFacts`, `ingredients`, `bestPrice`. **Добавить:** `gtin` (GTIN-14) как
  главный join-ключ; `productType` (BASE|ASSEMBLY).
- `DonorOffer` (ЕСТЬ) — цена по ритейлеру: `retailer`, `price`, `via` (direct|instacart),
  `isFirstParty`, `fetchedAt`. Идемпотентно по `(retailer, retailerProductId)`. Это твоя провенанс+время.
- `RetailPrice` (УПРАЗДНИТЬ в вышеупомянутые) — дубликат DonorProduct/DonorOffer с другим ключом.
- Каждый сырой факт несёт: `price`, `imageUrls` (все фото, с ролью main/nutrition/ingredients/
  infographic), `title`+`description`+`bullets` (весь текст), `attributes`, `fetchedAt`,
  `confidence`, `matchMethod` (gtin|title|llm|manual).

**СЛОЙ 2 — OURS («что сейчас живо на маркетплейсе»)** → `WalmartCatalogItem` + `AmazonListingSnapshot`.
Зеркала маркетплейса (обновляются синком). Правда про НАС, не про товар. Скоры «что чинить» —
в `WalmartListingQualityItem` / `AmazonListingHealthItem`.

**СЛОЙ 3 — GENERATED («что предлагает AI, по политике маркетплейса + brand-voice»)** → `GeneratedContent` + `MasterBundle`/`BundleComponent`.
Версионированные черновики. Читают Слой 1 + Слой 2 как вход; **сами ничего не добывают.**
На публикации становятся Слоем 2.
- `GeneratedContent` (ЕСТЬ, расширить) — `title`, `bullets`, `description`, `attributes`,
  `mainImageUrl`, `imageUrls`. **Добавить:** `policyProfile` (amazon-grocery|walmart-mp),
  `brandVoiceVersion`, `sourceFactIds` (какие сырые факты кормили — трассируемость),
  `status` (DRAFT|APPROVED|PUBLISHED|BLOCKED). **Здесь живёт исправленная главная картинка**
  (N одиночных единиц) до публикации.

**Себестоимость (уже правильно, не трогать):** `SkuCost` (датированный true-cost, уже с
эталонным `@@unique([sku, source, effectiveDate])`), `SkuComponent` (BOM, ссылки на
`DonorProduct` по GTIN), `SkuShippingData` (вес/габариты/`unitsInListing`).

**Итого: 15 таблиц оставить, 3 расширить (DonorProduct, GeneratedContent, EnrichmentJob),
1 упразднить (RetailPrice → DonorProduct). Переписывания нет.**

## 4. Правило-закон, которое убивает столкновения

> **Модуль может только ЧИТАТЬ данные о товаре из каталога. Ровно ОДИН движок —
> `enrichProduct()` — может ПИСАТЬ сырые факты, и только ему разрешено ходить во
> внешний источник. Нет данных в каталоге → ответ «поставить в очередь обогащения»,
> а НЕ «сходить добыть здесь».**

- **Владелец добычи:** `enrichProduct()` в `src/lib/sourcing/`. Владеет всеми адаптерами
  источников (Unwrangle, SP-API, OpenClaw-браузер, Google) и всеми обходами стен
  (UPC-мультипак → `isBaseUnit`; бот-блок Walmart → Veeqo-фолбэк; Instacart +15%).
  **Каждая стена решается один раз, в одном месте, навсегда.**
- **Потребители** (картинки, COGS, создание листингов, репрайсер) зовут `enrichProduct({sku})`,
  получают кэшированные факты и читают. Кода добычи в них — **ноль**.
- **Анти-цикл (вот что кончает «повторные провальные прогоны»):** `enrichProduct`
  идемпотентен + кэширует. Перед любым сетевым вызовом проверяет Слой 1 на факты свежее
  `maxStaleness`. **Хватает кэша → ноль внешних вызовов.** И **одна очередь** (расширить
  `EnrichmentJob`) дедупит по `(productId, need)` — второй чат, попросивший то же,
  **прицепляется** к идущей задаче, а не запускает параллельный прогон. Три чата физически
  не могут начать три добычи.

Поток картинок становится: `enrichProduct({sku})` → взять фото базовой единицы из
`DonorProduct.imageUrls` → сложить `Sku.packSize` копий → записать `GeneratedContent.mainImageUrl`
→ аппрув человека → публикация в Слой 2. **Если фото базовой единицы не в кэше — ставит в
очередь и СТОП, браузер не открывает.** Эта дисциплина и есть всё лекарство.

## 5. Порядок сборки (самый первый маленький шаг)

- **Фаза 0 (мельчайший первый шаг — сделать отдельно, зашипить):** добавить колонку
  **`gtin` (GTIN-14) как join-ключ** и забэкфилить для каждого `DonorProduct`, `SkuCost.sku`,
  `ChannelSKU.upc`, catalog-строк (нормализовать UPC-12/EAN-13 → 14 левым паддингом нулей —
  это убивает баги дедупа по ведущему нулю). *Одна колонка — недостающий join, чтобы любой
  чат находил данные другого.*
- **Фаза 1:** трёхуровневый мапинг `Product` (GTIN) → `Sku` → `Listing` поверх существующих
  таблиц. Теперь «pack of 6» и «pack of 2» сходятся в один товар.
- **Фаза 2:** обернуть все адаптеры источников в один `enrichProduct()` с проверкой кэша +
  идемпотентностью + confidence-гейтом. Потребителей пока не трогать.
- **Фаза 3:** унифицировать очередь — расширить `EnrichmentJob`, дедуп по `(productId, need)`.
- **Фаза 4:** включить закон — убрать любые внешние вызовы из кода потребителей; единственный
  вызыватель источников — `enrichProduct`. CI-проверка: «нет `fetch` к хостам ритейлеров/SP-API
  вне `src/lib/sourcing/`».
- **Фаза 5:** упразднить `RetailPrice` в `DonorProduct`; формализовать Слой 3.

Каждая фаза оставляет систему рабочей; big-bang нет.

## 6. ЧЕГО НЕ ДЕЛАТЬ (анти-паттерны, из-за которых падали прогоны)

- НЕ давать потребителю ходить во внешний источник напрямую. Нет в кэше → в очередь.
- НЕ ключевать идентичность по SKU. Ключ = GTIN-14. (SKU — это пак, а не товар.)
- НЕ заводить новую таблицу товара под новую фичу. Расширять каталог.
- НЕ перезаписывать один слой другим (raw ≠ ours ≠ generated). Дописывать с провенансом.
- НЕ верить Walmart-UPC как идентичности производителя — это seller-assigned мультипак-коды;
  матчить к базовой единице по title, флажок `isBaseUnit`.
- НЕ пере-добывать товар, который два чата уже добыли — сперва проверь кэш/свежесть.
- НЕ делать big-bang переписывание. Фаза 0 — одна колонка; шипи её первой.
- НЕ докладывать статус товара/листинга из нашей БД как живую правду — Слой 2 это *зеркало*;
  сверять с SP-API / Walmart перед действием.

---

**Одним абзацем:** есть одна реальная вещь — товар, опознаваемый по штрихкоду (GTIN), а не
по твоему SKU. «Pack of 6» и «pack of 2» — два SKU *одного* товара, поэтому его данные
хранятся один раз и каждый размер пака их наследует. Вокруг товара — три подписанные полки:
**RAW** (что говорит мир — каждое фото, весь текст, все цены, со штампом где и когда взяли),
**OURS** (что живо на каждом маркетплейсе сейчас), **GENERATED** (что предлагает AI, переписанное
под правила маркетплейса и твой brand-voice, включая исправленную главную картинку). Ровно
**один** движок ходит в интернет и наполняет полку RAW, и он помнит всё, чтобы не тянуть
одно и то же дважды. Картиночник, калькулятор себестоимости и писатель листингов — просто
читатели этих полок. Три чата ходили по кругу, потому что каждый был своим добытчиком со
своей полкой; заставь их читать один каталог и дай одному движку всю добычу — и круги кончатся.
