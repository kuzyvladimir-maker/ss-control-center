# 🏛️ Архитектура каталога товаров (КАНОН)

> **Зачем:** единый лаконичный ответ «как правильно построить решение», который
> **прекращает круги** между тремя чатами (картинки / COGS / создание листингов).
> Модель согласована с владельцем 2026-07-05 и сверена с живой схемой
> `ss-control-center/prisma/schema.prisma`. Все три чата равняются на ЭТОТ документ.
> Связано: [[product-sourcing-engine]], [[reference-catalog-engine]],
> [[walmart-ideal-listing-spec]], [[task-registry]]. Память: `project_reference_catalog`,
> `project_cogs_catalog_pricing_roadmap`, `reference_retail_source_capabilities`,
> `reference_buying_retailers` (зона закупки ZIP 33765).

## Суть за 30 секунд

**Два каталога + рецепт, который их связывает:**
1. **Справочный каталог товаров** — независимые реальные товары мира (любой бренд/
   ритейлер), доступные **в радиусе моего ZIP** (нет локально → не нужен, я перепродаю).
2. **Каталог моих SKU/листингов** — то, что я продаю на маркетплейсах.
3. **Связь = РЕЦЕПТ (состав) SKU** — список «**вариант товара × количество единиц**».
   Не GTIN. GTIN — уровнем ниже (как купить пак).

Себестоимость и контент листинга **вытекают из рецепта**: движок берёт каждую строку
рецепта → находит товар в справочнике → цену/фото/текст → складывает.

## 1. Корень кругов — одним предложением

**Три чата сталкиваются, потому что ни одна таблица не владеет «правдой о товаре»:**
каждый чат завёл свой кусок со своим ключом и своим кодом добычи, поэтому они не видят
результатов друг друга — каждый заново тянет те же данные, бьётся в ту же стену и падает
одинаково. В схеме одни и те же факты лежат в **7 таблицах без общего ключа** (`RetailPrice`,
`DonorProduct`, `WalmartCatalogItem`, `AmazonListingSnapshot`, `GeneratedContent`,
`ChannelSKU`, `MasterBundle`). Семь копий — семь добытчиков — семь стен. Это и есть
«повторные провальные прогоны».

## 2. Модель — одна картинка

```
  СПРАВОЧНЫЙ КАТАЛОГ ТОВАРОВ                МОИ SKU / ЛИСТИНГИ
  (независимые товары в радиусе ZIP)       (то, что продаю)

  Вариант (концепт)                         SKU  { marketplace, контент, себестоим. }
   "Uncrustables Strawberry"                  │
      │ 1 вариант → много паков                │ рецепт = состав
      ▼                                        ▼
  Пак (как купить)                          строки рецепта:
   "коробка 10, GTIN X, $Y, есть в ZIP"      [ Вариант × кол-во единиц ]
   "коробка 4,  GTIN Z, $W, есть в ZIP"
      ▲                                        │
      └──────── рецепт ссылается на ВАРИАНТ ───┘
                (по нормализованному имени, НЕ по GTIN)
```

**Справочный каталог — двухуровневый:**
- **Вариант (концепт)** — «Uncrustables Strawberry». Ключ = нормализованное имя
  (бренд + линия + вкус). **Рецепт ссылается СЮДА.**
- **Пак (покупаемый)** — конкретная фасовка под вариантом: коробка N штук, свой GTIN,
  своя локальная цена, есть ли в радиусе ZIP. У варианта их может быть несколько
  (разные коробки). **GTIN живёт здесь**, на уровне «как купить».

**Мой SKU — то, что я продаю:** поля `marketplace` (чей SKU), `рецепт` (состав),
себестоимость, свой контент. **Рецепт — поле/подтаблица SKU**, третий каталог не нужен.

**Почему связь = рецепт, а не GTIN:** листинг ≠ один товар. Мультипак = N одного варианта;
бандл/гифт-сет = разные варианты. Один GTIN не может это выразить. **Рецепт — единая форма
для всего** (у мультипака просто одна строка). А один вариант = несколько GTIN (разная
фасовка) — поэтому рецепт держит ВАРИАНТ, а движок сам подбирает пак.

## 3. На реальных таблицах (не переписываем — расширяем)

**Справочный каталог:**
- **`DonorProduct`** (ЕСТЬ) = **Вариант**. Уже есть `identityKey` (норм. имя), `title`,
  `description`, `bullets`, `attributes`, `imageUrls`, `nutritionFacts`, `ingredients`.
  Ключ рецепта = `identityKey`. `gtin`/`upc` тут — вспомогательные для дедупа.
- **`DonorOffer`** (ЕСТЬ) = **Пак**. `retailer`, `retailerProductId`(=GTIN/itemId), `price`,
  `via` (direct|instacart), `isFirstParty`, `fetchedAt`. **Добавить:** `packSize` (штук в
  коробке), `availableInZip` (есть в радиусе). Идемпотентно по `(retailer, retailerProductId)`.
- `RetailPrice` (УПРАЗДНИТЬ в `DonorProduct`+`DonorOffer` — это их дубль с другим ключом).

**Мой каталог:**
- **`ChannelSKU`/`Sku`** (ЕСТЬ) = мой SKU. **Убедиться, что есть поле `marketplace`.**
- **`SkuComponent`** (ЕСТЬ!) = **РЕЦЕПТ.** Каждая строка = `(sku, → DonorProduct вариант, qty)`.
  Это и есть связь моего SKU со справочником. Мультипак = одна строка; бандл = много.
- **`SkuCost`** (ЕСТЬ, эталон `@@unique([sku, source, effectiveDate])`) = себестоимость,
  считается из рецепта: Σ по строкам (qty × самый дешёвый пак варианта в ZIP).
- **`SkuShippingData`** (ЕСТЬ) — вес/габариты/`unitsInListing`.
- **`WalmartCatalogItem` / `AmazonListingSnapshot`** (ЕСТЬ) — зеркала «что живо на
  маркетплейсе сейчас» (правда про НАС, обновляется синком).
- **`GeneratedContent`** (ЕСТЬ, расширить) — черновики контента, что публикуем (title/
  bullets/description/attributes + **исправленная главная картинка N единиц**). **Добавить:**
  `policyProfile`, `status` (DRAFT|APPROVED|PUBLISHED), `sourceFactIds` (из каких вариантов
  собрано). Читает справочник + зеркало; сам ничего не добывает; на публикации → в зеркало.

**Итого: оставить, расширить `DonorOffer`+`GeneratedContent`, упразднить `RetailPrice`. Переписывания нет.**

## 4. Как всё вытекает из рецепта (пример Uncrustables)

Листинг-микс, рецепт = `[Strawberry×10, Grape×10, Raspberry×10]`:
- **Себестоимость** = для каждой строки движок под вариантом берёт самый дешёвый набор
  коробок в ZIP, чтобы набрать 10 (напр. 1×коробка-10 или 2×коробка-5) → сумма трёх строк.
- **Контент/фото** = из вариантов справочника (фото каждого вкуса, тексты, атрибуты) →
  `GeneratedContent` собирает главную картинку и текст по политике маркетплейса.

Мультипак `[Strawberry×6]` и бандл — та же механика, другое число строк.

## 5. Закон, который убивает столкновения

> **Модуль только ЧИТАЕТ из каталога. Ровно ОДИН движок — `enrichProduct()` — ПИШЕТ
> справочник (варианты+паки) и только ему разрешено ходить во внешний источник. Нет в
> каталоге → «поставить в очередь», а НЕ «сходить добыть здесь».**

- Движок владеет всеми адаптерами (Unwrangle, SP-API, Google/CSE, OpenClaw/CDP-браузер) и
  всеми обходами стен (мультипак→`isBaseUnit`, бот-блок Walmart→Veeqo, ZIP-фильтр). Каждая
  стена решается один раз.
- Идемпотентность + кэш: перед сетевым вызовом проверяет свежесть в справочнике; хватает →
  ноль вызовов. Одна очередь (`EnrichmentJob`) дедупит по `(variant, need)` — второй чат
  **прицепляется**, а не запускает второй прогон. **Три чата физически не начнут три добычи.**

## 6. Порядок сборки (мельчайший первый шаг)

- **Фаза 0:** для листингов-мультипаков этого чата **проставить рецепт** — каждому SKU строка
  `SkuComponent` `(sku → вариант DonorProduct, qty=packSize)`, сматчив по нормализованному
  имени. Это **тот самый недостающий join**: сразу даёт и фото (вариант→imageUrls→тайл), и
  себестоимость. Мелко, безопасно, полезно всем 3 чатам.
- **Фаза 1:** нормализовать `DonorProduct.identityKey` (вариант) + `DonorOffer.packSize`.
- **Фаза 2:** обернуть все адаптеры в один `enrichProduct(variant)` с кэш-проверкой.
- **Фаза 3:** унифицировать очередь (`EnrichmentJob`, дедуп по `(variant, need)`).
- **Фаза 4:** включить закон — убрать внешние вызовы из потребителей; CI-проверка «нет
  `fetch` к ритейлерам/SP-API вне `src/lib/sourcing/`».
- **Фаза 5:** упразднить `RetailPrice`; формализовать `GeneratedContent` (status/policy).

## 7. ЧЕГО НЕ ДЕЛАТЬ

- НЕ давать потребителю ходить во внешний источник. Нет в кэше → в очередь.
- НЕ ключевать рецепт по GTIN. Рецепт держит **вариант**; GTIN — уровень пака.
- НЕ считать листинг = один товар. Листинг = **рецепт** (может быть много вариантов).
- НЕ заводить новую таблицу товара под фичу. Расширять справочник.
- НЕ перезаписывать справочник черновиками (`GeneratedContent` ≠ `DonorProduct`).
- НЕ верить Walmart-UPC как идентичности — это seller-мультипак-коды; матчить к варианту
  по имени, флажок `isBaseUnit`.
- НЕ пере-добывать вариант, который уже в каталоге — сперва проверь свежесть.
- НЕ big-bang. Фаза 0 — проставить рецепт мультипакам; шипи первой.
- НЕ докладывать статус из нашей БД как живую правду — зеркало сверять с SP-API/Walmart.

---

**Одним абзацем для владельца:** есть два каталога. Первый — **справочник товаров**: реальные
вещи мира, которые можно купить рядом со мной; у каждого варианта («Uncrustables Strawberry»)
своя инфа и свои паки-коробки с ценами. Второй — **мои SKU/листинги** на маркетплейсах. Связь
между ними — **рецепт**: каждый мой листинг = список «вариант × сколько единиц» (мультипак =
одна строка, бандл/гифт-сет = несколько). Из рецепта сами вытекают себестоимость (сумма самых
дешёвых коробок под каждый вариант в моём ZIP) и контент (фото/текст вариантов → главная
картинка N единиц). Добывает данные во внешний мир **один движок** и кэширует навсегда;
картиночник, калькулятор себестоимости и писатель листингов — просто читатели этих двух
каталогов. Круги были потому, что каждый чат был своим добытчиком со своей копией; один
справочник + один добытчик + рецепт-связь — и круги кончаются.
