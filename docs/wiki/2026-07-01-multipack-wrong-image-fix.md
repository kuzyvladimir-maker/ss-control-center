# Multipack Optimizer — исправление «чужих/дублирующихся» главных фото (2026-07-01)

> Инцидент и фикс движка мультипак-ремедиации. Владелец заметил на галерее
> «свежих 50», что на разных листингах стоит одно и то же фото не того товара.
> Связано: [[2026-06-15-walmart-optimizer-wave1-session]],
> [[reference-catalog-engine]], [[sourcing-engine]]. Память:
> `project_walmart_remediation_worker`, `project_walmart_quantity_confusion`.

## Симптом
На галерее fresh-50 главные фото были неверными: у *Sara Lee Artesano*, *Jewish
Rye*, *Hot Dog Buns* и др. стояла сетка из **Pepperidge Farm «8 Soft White»
Hamburger Buns**. 47 тайлов схлопнулись в **30 уникальных**; один чужой файл стоял
на 6 листингах (байт-в-байт). Всё было **SUBMITTED в Walmart** (не превью).
Текст (title/bullets/description) при этом был корректный — неверны только фото.

## Первопричина (3 слоя)
1. **`enrich.ts` — гейт совпадения донора по 2 словам.** Матч шёл по первым двум
   токенам названия = только бренд («Pepperidge Farm», «Sara Lee»). Любой товар
   бренда проходил и писался в `RetailPrice` под наш SKU → пул одного листинга
   раздувался до 10-21 чужого товара / до 172 фото. Плюс fail-open («нет
   совпадений → верни всё»).
2. **`vision.ts` picker брал «самый чистый фронт» из мешанины** — раз за разом одну
   и ту же глянцевую генерик-картинку. Отсюда одинаковые фото на разных SKU.
   Отдельно: DONOR-FIRST shortcut возвращал `preferUrl` (первое фото пула) **без
   проверки варианта**; вариант-фильтр при неоднозначности делал `keep all`.
3. **Предохранителя идентичности не было.** `verifyMainImage` проверял только «N
   вертикальных фронтов на белом», НЕ «тот ли это товар». Чужой-но-чистый тайл
   проходил. Регрессия «ужесточённого» алгоритма (выбор по всему пулу); старые
   батчи 14-18 июня побайтовых дублей не давали.

## Фикс
- **`enrich.ts`**: гейт теперь требует бренд **И** долю вариант/тип-токенов
  (`GATE_STOP` отсеивает pack/size/unit-шум; порог ≥50% вариант-токенов, мин.1),
  **fail-closed** (пустой пул лучше замусоренного — пустой = do-no-harm SKIP).
- **`vision.ts`**: (а) новый `frontMatchesListing(url,title)` — строгий
  identity-гейт (бренд+тип+вариант по тексту этикетки), **fail-closed** на
  ошибке/неуверенности; (б) DONOR-FIRST shortcut отключён, когда передан
  `listingTitle`; (в) вариант-фильтр больше не «keep all» — при отсутствии
  совпадения возвращает null.
- **`remediate.ts`**: единый choke-point `tileVerifiedMain(donorUrl)` = identity →
  tile → verify. Применён ко ВСЕМ путям (strict / rescue / deep-enrich). `keep`-путь
  теперь оставляет текущее фото только если оно И структурно ок, И проходит
  identity (иначе не «залипает» на прежнем неверном тайле).

## Проверка (без записи в Walmart)
Прогон исправленных функций по 47 fresh-50 над существующими пулами:
**identity-гейт заблокировал 100% чужих доноров** (в логе — точные причины:
«brand is Wonder, not Pepperidge Farm», «Milano cookies», «Sara Lee vs Arnold»…).
Починено из наличного пула **11/47** (11 уникальных доноров — коллапса нет; напр.
1176 → корректный Jewish Rye Seeded ×4). Остальным **36/47** правильного фото в
пуле нет → нужен enrich (BlueCart/Unwrangle) по полному названию.

## Последствия для эксплуатации
- Движок теперь **fail-safe**: неверное фото не публикуется никогда (при
  неуверенности главное фото не трогается). Это разблокирует большой прогон
  БЕЗОПАСНО (он просто пропустит фото, которые не может подтвердить).
- Чтобы РЕАЛЬНО ставить правильные фото (и починить 36/47), нужен реактивированный
  BlueCart/Unwrangle — enrich по полному названию с уже строгим гейтом.
