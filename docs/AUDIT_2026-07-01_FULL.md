# Полный технический аудит SS Control Center — 2026-07-01

Проведён многоагентный аудит (43 агента, ~3.5 млн токенов, 1298 обращений к коду).
11 агентов картографировали модули, 8 искали проблемы по направлениям, каждая
серьёзная находка перепроверена отдельным агентом-скептиком, финальный агент-критик
искал пропуски. Три топ-находки по безопасности перепроверены вручную.

Масштаб: 879 файлов TS/TSX, 278 API-роутов, 53 страницы, 105 моделей БД, 28 cron-задач,
202 скрипта. TypeScript компилируется чисто. ESLint: 263 замечания (в основном `any`).

## 🔴 СДЕЛАТЬ СЕГОДНЯ (безопасность — проверено вручную)

### 1. В git лежит рабочая база `dev.db` с 5 живыми Google-токенами — КРИТИЧНО
`ss-control-center/dev.db` и `ss-control-center/prisma/dev.db` закоммичены. В таблице
Setting — 5 refresh-токенов Google OAuth (по 103 символа) для всех Gmail-аккаунтов
5 магазинов. Refresh-токен Google не истекает сам. Доступ к почте = сброс паролей
маркетплейсов. **Действия:** (1) немедленно отозвать/пере-выпустить все 5 токенов в
Google; (2) `git rm --cached dev.db prisma/dev.db`, добавить `*.db` в .gitignore;
(3) вычистить из истории git (git filter-repo / BFG).

### 2. Публичные `/api/debug/*` меняют живые заказы Veeqo — verified
`src/proxy.ts:65` пропускает `/api/debug/*` вообще без авторизации («временно»,
с 2026-05-03). Роут `veeqo-tag-test` ПИШЕТ тег в реальный заказ Veeqo — тег «Placed»
означает «уже куплено», любой из интернета может пометить некупленный заказ как
купленный → он не отгрузится → late shipment / A-to-Z. `veeqo-order` отдаёт полный
заказ с ПДн покупателя. **Действие:** удалить блок bypass в proxy.ts (строки 62–68)
и три роута в `src/app/api/debug/`.

### 3. Cron-авторизация «fail-open» — verified
Все 29 cron-роутов проверяют секрет как `if (process.env.CRON_SECRET && ...)` — если
переменная не задана (перенос проекта, preview-окружение), проверка пропускается и
репрайсер / ship-confirm становятся публично вызываемыми обычным GET. **Действие:**
единый `requireCronAuth()`, который при `VERCEL===1` и отсутствии секрета возвращает
401, а не пропускает.

**Прочий долг по безопасности (не срочно):** пароли на одном SHA-256 (нужен
bcrypt/argon2); RBAC fail-open в `proxy.ts:116` (нет cookie → доступ разрешён);
единый статический bearer-токен = полный админ на всех 278 роутах; на логине нет
rate-limit; destructive Walmart-эндпоинты (retire, cancel, label-buy) без авторизации.

## КАРТА ПРОЕКТА

**Что это:** веб-командный центр для арбитража продуктов питания на Amazon (5 акк.)
и Walmart (1 акк.). Единый пульт: заказы, доставка (замороженное!), закупки, клиентский
сервис, здоровье аккаунтов, финансы, репрайсинг и «фабрика листингов» Bundle Factory.

| Модуль | Состояние | Суть | Главная проблема |
|---|---|---|---|
| Shipping Labels | ✅ рабочий | покупка этикеток, замороженная логика, Monday-shift | Monday-shift иногда пишет дату на неделю позже реально расквоченной |
| Bundle Factory | 🟡 частичный | авто-фабрика листингов-корзин, публикация на Amazon доказана E2E | публикация на паузе: пул UPC пуст (все 2996 в карантине после коллизии) |
| Amazon Listing Health | ✅ рабочий | свой Listing Score, ремедиация, LLM-советник | rescore после фиксов — тихий no-op; свип каждые 6ч затирает enriched-метрики |
| Amazon Core Sync | ✅ рабочий | SP-API, синк заказов, adjustments, отчёты | orderTotal не обновляется → Pending-заказы навсегда $0, занижают выручку |
| Walmart | ✅ рабочий | заказы, ship-confirm, ремедиация, фабрика мультипаков | дрейф схемы (колонка `attempts` без миграции); ship-confirm ложно-Shipped |
| Customer Hub | 🟡 частичный | AI-обработка сообщений (полноценно) + A-to-Z/feedback (заглушки) | нет крона-синка + окно Gmail 2 дня → сообщения теряются; auto-resolve закрывает жалобы без ответа |
| Account Health | 🟡 частичный | AHR/ODR/политики + adjustments + frozen-аналитика | дневной крон физически не успевает забрать свой отчёт → снапшоты почти не появляются |
| Procurement | ✅ рабочий | список закупок + reference-catalog с воркерами | harvest-крон крутит платные ретраи вечно; priority-крон готов, но не в расписании |
| Finance / Economics | ✅ рабочий | Funds/P&L/COGS/репрайсер | распределение денег не атомарно (двойное начисление при сбое); репрайсер невидим |
| Core Infra | ✅ рабочий | auth, RBAC, Prisma, Telegram, Jackie MCP | см. раздел безопасности |
| Wiki / Backlog | ✅ рабочий | живая память проекта (976 статей) | ~30 замороженных задач; кредиты Anthropic кончились 2026-07-01 |

## 🟠 БАГИ, КОТОРЫЕ СТОЯТ ДЕНЕГ ИЛИ АККАУНТОВ

1. **Замороженное: Monday-shift пишет дату на неделю позже** — `plan/route.ts:1146`
   квотит на `nextMondayFrom(today)`, а сохраняет `nextMondayFrom(labelDate)`. При
   рефреше после 15:00 в пятницу карточка скажет «отгрузить в понедельник +7».
   Оттаявшая еда, возвраты, удар по ODR. Фикс: переиспользовать уже вычисленный Monday.

2. **Двойная покупка этикеток** — Veeqo-buy (`buy/route.ts:139`) может перекупить уже
   купленный item (нет проверки `status!=="bought"`, нет атомарного claim); Walmart-buy
   (`walmart/buy/route.ts:105`) — check-then-act: два параллельных запроса оба покупают.
   Прямая потеря денег в самой частой ежедневной операции. Фикс: атомарный claim через
   `updateMany` + unique-constraint 409.

3. **Заказы Pending навсегда остаются $0** — `orders-sync.ts:74` в update не пишет
   orderTotal. Amazon не отдаёт сумму для Pending, а большинство заказов впервые
   ловятся именно в Pending → выручка на дашборде занижена. Фикс: писать orderTotal
   в update + разовый ре-синк за 30 дней.

4. **Дневной крон здоровья Amazon не забирает свой отчёт** — `account-health-amazon`
   опрашивает 2 раза за 3 секунды, отчёт готовится 30с–3мин → снапшоты появляются
   только когда Владимир вручную жмёт «Обновить». Фикс: `maxDuration=300` + цикл
   опроса каждые 15с.

5. **Репрайсер: кэп 10% на РАН, но кронов 12/сутки** — `reprice-engine.ts:36`
   демпер может уронить цену ~10% двенадцать раз в день; SKU без SkuCost падают к
   полу $1. Плюс комиссия захардкожена 15% flat, хотя grocery ≤$15 = 8% → репрайсер
   отказывается от выигрышных Buy Box. Плюс RepliceLog никто не читает — репрайсер
   меняет цены вслепую. Фикс: кумулятивный дневной кэп, тиерная комиссия, страница лога.

6. **Распределение денег не атомарно** — `finance/run.ts:73`: балансы фондов
   инкрементятся до пометки payouts как distributed; сбой в середине → повторный ран
   раздаёт те же деньги снова (инфляция балансов); два клика «Distribute» = двойная
   раздача. Фикс: `prisma.$transaction` + идемпотентность.

7. **Ship-confirm Walmart подтверждает по двум origin-скана** — `walmart-ship-confirm:77`
   считает движением любые ≥2 события, а USPS штатно даёт 2 pre-shipment события →
   заказ помечается Shipped, пока коробка на полке. Бьёт по OTD (критерий Buy Box).

8. **Customer Hub теряет сообщения** — синк только вручную, окно Gmail `newer_than:2d`;
   любой перерыв >2 дней = навсегда пропущенные письма при дедлайне ответа 24ч. Плюс
   auto-resolve закрывает повторную жалобу без ответа. Фикс: крон каждые 3ч + окно 7д +
   защита auto-resolve.

9. **A-to-Z `shippedOnTime` всегда true** — `atoz/route.ts:33` сравнивает окно отгрузки
   само с собой. Значок «Claims Protected» показывается ошибочно → неверная маршрутизация
   «кто платит».

10. **Fire-and-forget после ответа на Vercel** — `product-type/route.ts:66` и др.:
    `void (async…)` после return убивается рантаймом → Veeqo-тег не ставится,
    `syncedToVeeqo` навсегда false. Фикс: `await` или `waitUntil()`.

**Класс «часовая бомба»:** ~45 мест берут бизнес-дату через сырой
`.toISOString().slice(0,10)` вместо канонов `todayNY`/`utcToPacificYMD` (нарушение
собственного правила проекта). Даёт сдвиг на день в дедлайнах, погоде, отчётах.

## 🧟 ЗОМБИ И ЗАСТРЯВШИЕ ЗАДАЧИ

**Очереди без «жнеца» (застревают навсегда):**
- `AmazonRemediationQueue` — строки RUNNING зомбируются (нет reaper), потом
  auto-improve вносит SKU в вечный blacklist; claim не атомарный → двойная запись
  в живые листинги Amazon.
- Отчётные машины (Amazon growth + Walmart Buy Box) — если запрос отчёта падает,
  строка вечно IN_PROGRESS, новый отчёт НИКОГДА не запрашивается → данные тихо
  замирают.
- `reference-harvest-worker` — упавший продукт не двигает updatedAt → ретрай каждые
  5 минут вечно, жжёт платные кредиты Unwrangle/BlueCart и может заморить бэклог.
- Свипы курсоров (Walmart LQ / Amazon Health) — протухший курсор пере-сохраняется
  на каждой ошибке → свип не завершается, удалённые листинги не чистятся.
- `tickBatch` Bundle Factory — крон и браузер тикают один батч без claim → двойная
  генерация + двойной расход Claude.
- `UPCPool` — RESERVED-строки текут навсегда (TTL пишется, но никто не читает);
  каждый крэш между reserve и create безвозвратно теряет платный штрихкод.

**Готовые, но не подключённые кроны (фича написана, но не работает):**
- `procurement-priority` — Telegram-нудж по приоритетным заказам, убрали на Hobby,
  не вернули на Pro.
- `pricing-sync` — монитор дрейфа цен Uncrustables, никогда не в расписании.
- `bundle-factory/poll-pending` — поллер статуса публикации + UPC self-heal, не в
  расписании → SUBMITTED-листинги вечно «в процессе».
- `frozen-analysis` — наоборот, дублируется (Vercel-крон + n8n одновременно).

## ⚡ УСКОРЕНИЕ СЕРВИСА

Turso — сетевая БД: каждый запрос = round-trip (десятки мс). Главный рычаг —
сократить число запросов на страницу и перестать дёргать Veeqo вслепую.

**Бэкенд (по убыванию боли):**
1. **Сайдбар опрашивает 2 самых тяжёлых Veeqo-эндпоинта каждые 60с на КАЖДОЙ странице**
   (`SidebarContent.tsx:386`). 2–3 вкладки = 4–6 полных скрейпов Veeqo в минуту в фоне
   → сам себе 429. Фикс: лёгкий `/api/sidebar/counts` только из БД, интервал 5 мин,
   пропуск при `document.hidden`.
2. **`/api/veeqo/orders` игнорирует `page_size`** — для виджета на 8 строк тянет ВЕСЬ
   заказный журнал Veeqo. Фикс: честно брать одну страницу.
3. **Кнопка «Обновить» (`/api/sync`)** — ~750 последовательных upsert по 5 магазинам +
   финансы + awaited fan-out → минуты. Фикс: `createMany` + батчи + Promise.all по магазинам.
4. **`/api/sales-overview`** — тянет полные строки заказов (с `rawData` blob) без `select`,
   дважды, + живой скрейп до 5000 заказов Veeqo. Фикс: `select` только нужных полей + кэш.
5. **Ингест Sales&Traffic** — ~2000 последовательных запросов Turso на магазин внутри
   120с-крона каждые 20 мин. Фикс: один `findMany(in)` + батч-транзакции.
6. **Нет кэша нигде** (0 из 278 роутов) — каждый хит дашборда пересчитывает всё. Фикс:
   `unstable_cache` c TTL 60–120с + `revalidateTag` из кронов.
7. **Нет индексов** на горячих where: `SyncLog` (вообще нет), `AmazonOrder.status/lastUpdateDate`,
   `ShippingAdjustment.createdAt`, `AtozzClaim`. Плюс таблицы-логи растут без ретеншена.

**Фронтенд:**
- Открытие `/shipping` = 4 полных скрейпа Veeqo одновременно (страница+сайдбар+mergeable+summary)
  и неограниченный веер Walmart-квот → сам себе 429, кнопка Buy гаснет. Фикс: пул на 3–4.
- `/shipping/page.tsx` — 6360 строк в одном 'use client', `OrderRow` (~900 строк) без
  `React.memo` → пере-рендер всего на каждое нажатие клавиши. Фикс: вынести + memo.
- `reference-catalog` опрашивает 8-запросный эндпоинт каждые 6с всё время enrichment.
- `economics` рендерит тысячи строк без пагинации; нигде нет `next/image` (миниатюры
  тянут оригиналы по 1–4 МБ).

## 💰 БИЗНЕС-РЕКОМЕНДАЦИИ, РИСКИ, ЧТО УПУЩЕНО

**Экзистенциальные риски (не только код):**
- Токены в git (см. §1) — фактически «ключи от бизнеса» лежат в открытом репозитории.
- **Нет бэкапов Turso / нет плана DR** — для не-технического владельца тихая потеря
  данных так же смертельна, как взлом. Настроить бэкап Turso + lifecycle R2.
- **Кредиты Anthropic кончились 2026-07-01** и тихо деградировали генерацию ВЕЗДЕ
  (Customer Hub, Bundle Factory, ремедиация) — 13 листингов ушли с fallback-текстом,
  нарушая твоё же правило. Нужен guard: крон следит за балансом → Telegram + авто-стоп.
- **Нет CI, нет тестов, нет мониторинга ошибок** — 28 денежных кронов могут падать
  молча. Минимум: GitHub Action (`tsc` + `eslint`) + обёртка `withCronLog` (пишет
  SyncLog + шлёт Telegram при throw).
- **Prompt-injection в Customer Hub** — движок читает текст покупателя и выбирает
  refund/replace; текст письма — контролируемый злоумышленником вход, никто это не
  фильтрует на границе отправки (`response-sender.ts:43` шлёт без ре-валидации).

**Что можно выжать (по business value):**
1. **Разблокировать Bundle Factory** — самая близкая к деньгам задача. Спека готова:
   импортировать пул Speedy (13 239 бесплатных UPC, CSV уже в docs/), добавить
   pre-check «свободен ли UPC на Amazon» в `reserveUpc`, собрать UPC Pool Manager UI.
   Одна сессия открывает весь revenue-движок фабрики.
2. **Закрыть петлю adjustments → Veeqo** — модуль уже считает корректный вес из
   поправок перевозчика; добавить кнопку «запушить исправленные габариты в каталог» →
   меньше reweigh-списаний.
3. **Показать репрайсер** — маленькая страница из RepriceLog (что снижено, где держим
   пол) вернёт видимость, потерянную при отключении Telegram-сводок.
4. **Авто-возврат-плохих-отзывов** — feedback-removal для carrier-fault ≤2★ (SP-API).
5. **Донор-фото в галерею Amazon** — уже собраны и в R2, но не публикуются
   (`other_product_image_locator_1..4`) — самый дешёвый рычаг конверсии.
6. **Живые цены закупки на карточках Procurement** — связать SKU с DonorProduct
   (UPC/identityKey) и показывать «дешевле всего сегодня: Walmart $2.48».

**Что упущено в мониторинге бизнеса:**
- `/analytics` net profit заблокирован: у AmazonOrder нет line items (нужен getOrderItems).
- Content/Returns/Brand-Analytics сигналы в Listing Health навсегда null → скор считается
  по 4 из 6 компонентов. Returns можно заполнить из уже собираемых adjustment-событий.
- ~450 листингов store1 вне мониторинга (лимит ~1000 у Listings API) — решается
  бесплатным отчётом GET_MERCHANT_LISTINGS_ALL_DATA.
