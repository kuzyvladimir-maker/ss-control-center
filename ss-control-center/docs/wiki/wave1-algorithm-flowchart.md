# Wave 1 — алгоритм по шагам (блок-схема)

Что делала Wave 1 с каждым листингом, от начала до конца. **★ = место дефекта**
(выбор/тайлинг неправильной картинки — обратная сторона / лайфстайл / промо).

## Уровень 1 — Воркер (Vercel Cron, каждые 2 минуты)

```
┌─────────────────────────────────────────────────────────────┐
│ CRON каждые 2 мин: /api/cron/walmart-remediation-worker       │
└───────────────┬───────────────────────────────────────────────┘
                │
   ① РЕАНИМАЦИЯ: зависшие 'running' > 10 мин  →  обратно в 'queued'
                │
   ② ФИНАЛИЗАЦИЯ отправленных фидов: спросить статус у Walmart
        ├─ PROCESSED ok=1            → 'done'  ✅
        ├─ конфликт каталога (QARTH) → пере-очередь "только картинка"
        └─ прочая ошибка             → 'error'
                │
   ③ БЮДЖЕТ-ГАРД: кредиты BlueCart < 300 → выключить обогащение
                │
   ④ БАТЧ-ГЕЙТ: держать ≤ 120 «в работе»; добрать из 'held'
                │
   ⑤ ВЗЯТЬ листинг из очереди (queued) → пометить 'running'
                │
                ▼   (передаём в обработку одного листинга, Уровень 2)
```

## Уровень 2 — Обработка ОДНОГО листинга (`buildAndSubmitOne`)

```
 1. GET товара у Walmart  →  UPC, productType, текущее имя
        └─ не найден ────────────────────────────► SKIP

 2. ОПРЕДЕЛИТЬ РАЗМЕР ПАКА (N)
        источник по приоритету: SkuShippingData → SkuCost
        → WalmartCatalogItem.titlePackCount → ListingQuality.titlePackCount
        └─ N < 2 (не мультипак) ─────────────────► SKIP

 3. ОБОГАЩЕНИЕ (ensureDonorImage)  [если включено бюджетом]
        нет фото в нашей базе?
        → BlueCart ПОИСК ПО НАЗВАНИЮ
        → сохранить В НАШУ БАЗУ (RetailPrice): цена, тайтл, описание, картинки

 4. ★ ВЫБРАТЬ БАЗОВУЮ КАРТИНКУ ДЛЯ ПЛИТКИ ★   ←─── ЗДЕСЬ ДЕФЕКТ
        берётся imageUrls[0] из RetailPrice (сорт. по packSize/confidence)
        ПРОБЛЕМА: это часто обратная сторона / лайфстайл / промо,
                  а НЕ чистое лицевое фото на белом фоне.
        └─ фото нет вообще ──────────────────────► SKIP "no donor photo"

 5. KB-ЗАХВАТ (fetchAndStoreDetail)  — НОВОЕ
        BlueCart ДЕТАЛИ → полная галерея, буллеты, описание,
        спецификации, ingredients, сырой JSON → сохранить в RetailPrice

 6. КОНТЕНТ (только если выбраны title/описание/буллеты)
        собрать заголовок/описание/буллеты + полировка Claude
        (в режиме «только картинка» — пропускается, Claude не вызывается)

 7. ★ СБОРКА КАРТИНКИ ★   ←─── ЗДЕСЬ ДЕФЕКТ ПРОЯВЛЯЕТСЯ
        composeTiledMainImage(базовая_картинка, N)
        = размножить базовую картинку в сетку N штук → загрузить в R2
        ПРОБЛЕМА: тайлит ту самую неудачную картинку из шага 4.
        (Раньше ещё генерился бейдж с текстом → шрифт не грузился →
         синие квадраты. Бейдж теперь УБРАН.)

 8. СОБРАТЬ ФИД MP_MAINTENANCE (частичный апдейт)
        отправляем только выбранные поля: главную картинку (+ контент,
        если выбран). `brand` НЕ шлём (вызывал конфликт QARTH).

 9. ОТПРАВИТЬ ФИД Walmart → получить feedId → пометить 'submitted'
        (+ записать "до"-метрики в лог)

        ▼
 10. ПОЗЖЕ (на следующем тике воркера) — Финализация (шаг ② выше):
        Walmart обрабатывает фид ~100 мин →
        PROCESSED ok → done; конфликт → ретрай "только картинка"; ошибка → error
```

## Где именно ломается

- **Шаг 4 + 7** — корень всех трёх типов брака на скринах:
  - Glory Foods → тайл из **обратной стороны** (Nutrition Facts).
  - Arnold Bread → тайл из **лайфстайл-баннера** (у этого товара в BlueCart
    вообще НЕТ чистого лицевого фото — только лайфстайл).
  - Pringles → тайл из **промо-арта**.
- Причина: на шаге 4 берётся «какая есть» картинка, без проверки, что это
  **чистый лицевой кадр товара на белом фоне**.

## Что это значит для фикса

Надёжно выбрать «чистое лицо» из галереи BlueCart **не всегда возможно**
(у части товаров его там нет). Поэтому варианты фикса:
1. Брать **оригинальную главную картинку самого листинга Walmart** (она была
   чистым лицевым кадром) и тайлить ЕЁ — самый надёжный источник.
2. Детектор «чистого фото» (белый фон) + правило «нет чистого → НЕ трогаем
   главную картинку» (не навреди).
3. Либо вместо плитки — чистый лицевой кадр + аккуратная подпись «Pack of N»
   (нужно починить шрифт в serverless).

Статус: Wave 1 **на паузе** до выбора стратегии фикса.

# Целевой алгоритм v3 — полная пересборка листинга (что строим)

Wave 1 (выше) только менял картинку. v3 — пересобирает листинг целиком из НАШЕЙ
базы знаний + правил Walmart, с проверкой приёма/отказа.

```
На каждый выбранный SKU:

A. ДАННЫЕ (из нашей базы — уже захвачены BlueCart-ом, шаг 3/5 Wave 1)
     все фото галереи, тайтл, буллеты, описание, nutrition/ingredients,
     спецификации, атрибуты, бренд, цена.

B. ПРАВИЛА (чего хочет Walmart — собираем в один «бриф»)
     • per-listing LQ issues (Walmart Insights API): что КОНКРЕТНО не так —
       "title formula", "nutrition_facts_label missing", "description too short"…
     • required attributes категории (Get Spec API): какие поля обязательны
     • наш brand-voice: без emoji / промо-прилагательных / health-claims
     • драйверы скора: Content & Discoverability

C. ГЕНЕРАЦИЯ (Claude, на входе A + B → закрывает flagged-issues)
     • Title  — формула Walmart (brand + ключевые атрибуты + размер) + "Pack of N"
     • Bullets — 3–7, фактические, закрывают content-issues
     • Description — нужной длины, фактическое
     • Attributes — nutrition_facts_label и др. недостающие (из захваченных данных)
     • ★ ГЛАВНАЯ КАРТИНКА: vision-модель выбирает чистое ЛИЦО (одна единица,
        белый фон) → если есть → плитка ×N (подписать N, в базу + R2);
        если чистого лица НЕТ → главную НЕ трогаем (не навреди)
     • Вторичные фото — реальная галерея

D. ФИД → ОТПРАВКА (MP_MAINTENANCE, scope-aware)

E. ПРИЁМ / ОТКАЗ (петля)
     • PROCESSED ok=1   → "OK"  + замер LQ-скор и конверсии ДО/ПОСЛЕ
     • отклонён (QARTH / карточка чужая/залочена) → "в доработку: catalog-locked"
                          → фолбэк: только картинка, либо флаг на ручную правку
     • прочая ошибка    → "в доработку: <причина>"
```

**Главные отличия от Wave 1:** (1) картинку выбирает ЗРЕНИЕ, а не «первая
попавшаяся»; (2) контент генерится под КОНКРЕТНЫЕ замечания Walmart, а не вслепую;
(3) явная пометка OK / в-доработку по результату приёма фида; (4) замер до/после,
чтобы видеть, помогает ли (после падения продаж это критично).
